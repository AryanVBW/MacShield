Privacy Policy

Extension Name: MacShield — Privacy Lock & Blur Developer: Aryan V (AryanVBW) Version: 3.1.0 Last Updated: April 6, 2026 Contact: GitHub Issues

1. Introduction

MacShield ("the Extension", "we", "our") is a browser privacy extension for Google Chrome built and maintained by Aryan V, an independent open-source developer. MacShield provides two core privacy features: a content blur system that obscures private messages and chat content across supported web platforms, and a website lock system that prevents unauthorized access to any website chosen by the user.

This Privacy Policy explains in complete detail what data MacShield accesses, what it stores, how it stores it, why it needs it, and what it absolutely does not do. We believe users have a right to understand exactly how a privacy tool works — especially one that touches sensitive areas like passwords and web content.

MacShield is fully open source. Every claim made in this Privacy Policy can be independently verified by reading the source code at:
https://github.com/AryanVBW/MacShield

2. Summary (Plain Language)

Before the legal detail, here is a plain summary of what MacShield does and does not do with your data:

What MacShield does What MacShield does NOT do
Stores your password as a hash locally Store your actual password anywhere
Saves your blur preferences locally Send any data to any server
Saves your list of locked sites locally Read, copy, or log your messages
Processes your password entry locally Share data with third parties

Everything MacShield stores lives exclusively on your own device inside Chrome's local extension storage. Nothing is transmitted anywhere.

3. Data We Access and Why

3.1 Website Hostnames

What: MacShield reads the hostname (domain name) of every page you visit — for example github.com, mail.google.com, or web.whatsapp.com.

Why: This is the core mechanism of both features. For the lock feature, MacShield checks the hostname against your list of locked sites at document_start (before the page renders) to decide whether to show the lock screen. For the blur feature, MacShield checks the hostname to decide whether to apply the blur CSS for supported platforms.

What we do NOT do: We do not record, log, store, or transmit the hostnames of pages you visit. The hostname check happens in memory only, in real time, and is discarded immediately. No browsing history is built or stored.

3.2 Authentication Information — Your Master Password

What: When you create a master password in MacShield, the extension processes that password locally to create a secure hash.

How it works technically:

  1. You enter a password in the popup or setup screen
  2. A cryptographically random 16-byte salt is generated using crypto.getRandomValues()
  3. Your password is concatenated with the salt and passed through SHA-256 via the Web Crypto API (crypto.subtle.digest)
  4. Only the resulting hash and the salt are stored in chrome.storage.local
  5. Your original password is never stored anywhere, in any form
  6. When you enter a password to authenticate, the same process runs and the resulting hash is compared — your entered password is then discarded from memory

What is stored:

  • ms_password_hash — the SHA-256 hash of your password + salt (a 64-character hex string)
  • ms_password_salt — the random salt used in hashing (a 32-character hex string)

What is NOT stored: Your actual password, in plain text or any reversible encoding, is never stored anywhere.

Where it is stored: Exclusively in chrome.storage.local on your own device. This storage area is sandboxed to the MacShield extension and cannot be read by websites or other extensions.

Who can access it: Only the MacShield extension itself. Not Anthropic, not Google, not any server, not any third party.

Why we need it: Without storing the hash, the extension cannot verify your password when you attempt to unlock a protected website. There is no alternative architecture that allows password verification without storing some representation of the password locally.

3.3 WebAuthn Credential ID (Touch ID)

What: If you choose to enroll Touch ID (on Mac) or device biometrics, the WebAuthn API creates a cryptographic credential through your operating system. MacShield stores only the credential ID — a non-sensitive reference identifier — not any biometric data.

How it works:

  1. You click "Enroll" in the Settings tab
  2. Chrome opens a WebAuthn popup (auth.html) which calls navigator.credentials.create() with authenticatorAttachment: "platform" and userVerification: "required"
  3. Your Mac's Secure Enclave or device TPM handles the biometric verification entirely — MacShield never sees your fingerprint or any biometric data
  4. The WebAuthn API returns a credential object — MacShield stores only the rawId (encoded as Base64) in chrome.storage.local under the key ms_webauthn_cred_id
  5. On future authentication, navigator.credentials.get() is called with the stored credential ID — your device handles biometric verification — MacShield receives only a success or failure signal

What is stored: A Base64-encoded credential ID string in chrome.storage.local

What is NOT stored: Any biometric data, fingerprint data, facial recognition data, or any data from your device's Secure Enclave. MacShield has no access to any of this — it is architecturally impossible for a browser extension to access Secure Enclave data.

Why we need it: The credential ID is required to construct the allowCredentials array when calling navigator.credentials.get() for authentication. Without it, the extension cannot request verification from the correct credential.

3.4 Locked Sites List

What: When you lock a website, MacShield stores that website's hostname in a local configuration object.

What is stored: A JSON object under the key ms_locked_sites in chrome.storage.local. Example structure:

{
  "github.com": true,
  "notion.so": true,
  "web.whatsapp.com": true
}

What is NOT stored: Any content from those websites, any data about what you do on those websites, or any history of when you visited them.

Why we need it: The lock guard script (lock-guard.js) runs at document_start on every page and must immediately know whether to show the lock screen. This requires the locked sites list to be readable from storage synchronously.

3.5 Blur Preferences

What: MacShield stores your blur settings so they persist between browser sessions.

What is stored in chrome.storage.local:

  • ms_blur_level — an integer (2–30) representing your chosen blur intensity in pixels
  • ms_blur_avatars — a boolean, whether profile picture blurring is enabled
  • ms_hide_mode — a boolean, whether full hide mode is active instead of blur
  • macshield_<hostname> — a boolean per supported platform, whether blur is enabled on that specific site (e.g. macshield_web_whatsapp_com: true)

What is NOT stored: Any message content, any chat history, any names or identifiers of people you communicate with, any images or media. The blur feature operates purely by applying CSS filter: blur() to page elements — it reads nothing from the page.

3.6 Session Unlock State

What: When you successfully authenticate (by password or Touch ID) to unlock a protected site, MacShield records that site as "unlocked for this session."

What is stored: A list of hostnames stored in chrome.storage.session under the key unlockedSites. Example: ["github.com", "notion.so"]

Key properties of session storage:

  • chrome.storage.session is cleared automatically when the browser is closed
  • It is never written to disk in a persistent way
  • It resets every browser session, meaning every time you open Chrome, all sites are locked again

Why we need it: Without session tracking, you would need to re-enter your password every single time you navigate to a new page on a locked site, which would make the extension unusable.

4. Data We Do NOT Access

The following is an explicit list of data that MacShield does not access, process, or store — ever:

  • Your message content — MacShield applies CSS blur to page elements without reading their text content. The blur is a visual filter only. No message text, email body, chat history, or communication content is ever read or stored by the extension.
  • Your browsing history — MacShield does not build, store, or transmit a record of which pages you visit. Hostname checks happen in memory only and are immediately discarded.
  • Your personal information — MacShield does not access your name, email address, phone number, physical address, date of birth, or any personally identifiable information.
  • Your location — MacShield does not access GPS coordinates, IP-derived location, or any location information.
  • Your financial information — MacShield does not access credit card numbers, bank account details, or any financial data.
  • Your health information — MacShield does not access any health or medical data.
  • Cookies or login sessions — MacShield does not read, modify, or delete cookies from any website.
  • Keystrokes or mouse activity — MacShield does not log keystrokes (outside of listening for the Alt+X shortcut key to toggle blur), mouse position, scroll position, or any user interaction data.
  • Photos, files, or downloads — MacShield does not access your file system, downloads, photos, or any local files.
  • Other extensions — MacShield cannot read data from other Chrome extensions.
  • Incognito mode data — MacShield does not request access to incognito windows and cannot see incognito browsing activity.

5. Data Sharing and Third Parties

MacShield shares absolutely no data with any third party, company, advertiser, analytics service, or government entity.

There are no third-party SDKs, analytics libraries, advertising networks, crash reporting services, or telemetry systems included in MacShield. The extension makes zero outbound network requests of any kind — with the sole exception of the optional Google Fonts stylesheet import used for the popup UI typeface, which is a cosmetic resource that carries no user data.

MacShield has no backend server, no database, no API, and no cloud infrastructure of any kind. There is no company behind MacShield — it is an independent open-source project.

6. Data Storage and Security

6.1 Where Data is Stored

All data is stored exclusively in Chrome's extension storage APIs:

  • chrome.storage.local — persistent across browser restarts, sandboxed to MacShield only
  • chrome.storage.session — cleared on browser close, sandboxed to MacShield only

These storage areas are not accessible to websites, other extensions, or any external entity.

6.2 How Passwords are Secured

MacShield uses the following security measures for password storage:

  • SHA-256 hashing via the Web Crypto API — a one-way cryptographic function
  • Random salt — 16 cryptographically random bytes generated per password, preventing rainbow table attacks
  • No plaintext storage — the original password exists in memory only during the brief moment of hashing, then is discarded
  • No transmission — the hash never leaves the device

6.3 How Touch ID is Secured

Touch ID authentication is handled entirely by the operating system through the WebAuthn standard. MacShield stores only a non-sensitive credential reference ID. All biometric processing occurs within the device's Secure Enclave or Trusted Platform Module, which is architecturally isolated from browser extensions. MacShield receives only a cryptographic assertion proving successful verification — it never has access to biometric data.

7. Permissions Explanation

MacShield requests the following Chrome permissions and uses them as follows:

  • storage — Required to save all user preferences and the password hash locally. Used exclusively for the data described in Section 3.
  • activeTab — Required so the popup can communicate with the content script on the current tab to toggle blur in real time.
  • tabs — Required to read the current tab's URL hostname and to broadcast lock/unlock state changes across multiple tabs of the same domain.
  • background — Required to run the service worker that manages password verification, session unlock tracking, Touch ID coordination, and lock badge display.
  • <all_urls> host permission — Required because the website lock feature must work on any domain the user chooses to protect. The lock guard script runs at document_start to prevent content flash before the lock screen appears. This permission does not grant MacShield the ability to read page content — it is used only to inject the lock screen CSS and script.
